# 並列処理とタスク分割

## はじめに

サブエージェントの最大の強みは「並列処理」にあります。このレッスンでは、Claude Codeがどのようにサブエージェントを使って並列作業を行うか、そして効果的な並列処理を引き出すための指示の書き方を学びます。

## 並列処理の基本概念

通常のClaude Codeセッションでは、タスクは直列に処理されます。

```
直列処理（サブエージェントなし）:
タスクA → タスクB → タスクC → 完了
合計時間: A + B + C

並列処理（サブエージェントあり）:
タスクA ─┐
タスクB ─┼→ 完了
タスクC ─┘
合計時間: max(A, B, C)
```

3つのタスクがそれぞれ10秒かかる場合、直列では30秒かかりますが、並列では約10秒で完了します。これがサブエージェントによる並列処理の本質的なメリットです。

## 並列ファイル検索の実例

大規模なコードベースで特定のパターンを検索する場合、サブエージェントによる並列検索が威力を発揮します。

### 単一エージェントによる検索（遅い）

```bash
> deprecated関数の使用箇所をコードベース全体で検索してください
# 一つのエージェントが順番にディレクトリを探索する
```

### 並列サブエージェントによる検索（速い）

```bash
> 以下を並列で調査してください:
> 1. src/components/以下でuseDeprecatedHookが使われている箇所
> 2. src/api/以下でlegacyFetchが使われている箇所
> 3. src/utils/以下でoldFormatDate関数が使われている箇所
> 4. src/pages/以下でclassComponentが使われている箇所
>
> 各ディレクトリを独立して調査し、結果をまとめてください
```

このように指示すると、Claude Codeは4つのサブエージェントを同時に起動し、それぞれが独立したディレクトリを調査します。

## 並列コード分析のパターン

### パターン1: ファイル単位の並列分析

```bash
> 以下のサービスファイルを並列で分析し、それぞれのエラーハンドリングを評価してください:
> - src/services/userService.ts
> - src/services/orderService.ts
> - src/services/paymentService.ts
> - src/services/notificationService.ts
>
> 各ファイルについて:
> - try-catchブロックの適切さ
> - エラーのログ出力
> - ユーザーへのエラーメッセージ
> を確認し、改善点をリストアップしてください
```

### パターン2: 機能横断的な並列チェック

```bash
> セキュリティ監査として、以下を同時に実行してください:
>
> 調査1: src/api/以下の全エンドポイントで認証チェックが実装されているか確認
> 調査2: src/models/以下でSQLインジェクション対策（パラメータ化クエリ）が使われているか確認
> 調査3: src/utils/以下でXSS対策（入力サニタイズ）が実装されているか確認
>
> 各調査は独立して進め、問題のある箇所をファイル名と行番号で報告してください
```

### パターン3: 異なる種類のタスクを並列実行

```bash
> 以下を並列で実行してください:
>
> タスクA: package.jsonを確認して、セキュリティ脆弱性のある依存ライブラリを特定する
> タスクB: .env.exampleを確認して、本番環境に必要な環境変数のドキュメントを作成する
> タスクC: src/types/以下の型定義ファイルを分析して、型の重複や不整合を特定する
>
> 3つのタスクが完了したら、それぞれの結果をまとめて報告してください
```

## 並列処理を指示するための表現

Claude Codeに効果的に並列処理を指示するためのキーワードと表現を覚えましょう。

### 明示的な並列化の指示

```bash
# キーワード: 「並列」「同時」「並行」
> 並列で以下のタスクを実行してください...
> 同時に以下の調査を行ってください...
> 並行して以下の作業を進めてください...
```

### 独立性を強調した指示

```bash
# 各タスクが独立していることを明示する
> 以下のタスクはそれぞれ独立しているため、同時に実行してください:
> - タスク1: ...
> - タスク2: ...
> - タスク3: ...
```

### サブエージェントを直接指定する指示

```bash
# サブエージェントの使用を明示的に促す
> サブエージェントを使って以下の作業を並列処理してください...
> 複数のエージェントを使って以下を調査してください...
```

## エージェントタイプとその特性

Claude Codeのエコシステムでは、異なる種類のエージェントがあります。それぞれの特性を理解することで、適切な使い方ができます。

### メインエージェント

ユーザーと直接対話するエージェントです。

- ユーザーからの指示を受け取る
- サブエージェントを起動・管理する
- 結果を統合してユーザーに返す
- 会話の全コンテキストを保持する

### Orchestratorエージェント（指揮者型）

複雑なタスクを分解し、複数のサブエージェントを管理することに特化したエージェントです。

```bash
# Orchestratorパターンの使用例
> 大規模なリファクタリングを計画・実行してください。
> まず全体像を把握し、作業を複数のサブタスクに分解してから、
> 各サブタスクを担当エージェントに割り当てて実行してください。
```

### Subagentエージェント（実行者型）

具体的な作業タスクを実行することに特化したエージェントです。

- ファイルの読み書き
- コードの検索と分析
- コマンドの実行
- 単一の明確なタスクの完遂

### 用途に応じた選択

| タスクの種類 | 推奨アプローチ |
|-------------|---------------|
| 単一ファイルの編集 | メインエージェント直接実行 |
| 複数ファイルの独立した分析 | 並列サブエージェント |
| 段階的な依存関係のある作業 | 直列サブエージェント（順序管理） |
| 広範なコードベース調査 | 並列サブエージェント + 結果統合 |
| 複雑な設計・アーキテクチャ判断 | メインエージェント（全コンテキスト必要） |

## 実践的な並列処理シナリオ

### シナリオ1: 新機能の並列実装

バックエンドAPIとフロントエンドコンポーネントを同時に開発します。

```bash
> ユーザーダッシュボード機能を並列で実装してください:
>
> サブタスク1（バックエンド）:
> - GET /api/dashboard エンドポイントを実装
> - ユーザーの統計情報（投稿数、コメント数、アクティビティ）を返す
> - src/api/routes/dashboard.ts に配置
>
> サブタスク2（フロントエンド）:
> - DashboardPage コンポーネントを作成
> - APIから取得したデータを表示するカードレイアウト
> - src/pages/Dashboard.tsx に配置
>
> サブタスク3（型定義）:
> - DashboardData インターフェースを定義
> - src/types/dashboard.ts に配置
>
> 各サブタスクは独立して実装できます。並列で進めてください。
```

### シナリオ2: 並列テスト生成

複数のサービスのテストを一度に生成します。

```bash
> 以下のサービスのユニットテストを並列で生成してください:
>
> - src/services/emailService.ts → tests/services/emailService.test.ts
> - src/services/smsService.ts → tests/services/smsService.test.ts
> - src/services/pushService.ts → tests/services/pushService.test.ts
>
> 各テストファイルには:
> - 正常系テスト（各メソッドの基本動作）
> - 異常系テスト（エラーケース、nullチェック）
> - モックの設定
> を含めてください。
>
> テストフレームワークはJestを使用してください。
```

### シナリオ3: 並列ドキュメント生成

複数のモジュールのドキュメントを同時に生成します。

```bash
> 以下のモジュールのAPIドキュメントを並列で生成してください:
>
> 対象: src/api/以下の全ルートファイル
> 形式: OpenAPI 3.0 YAML形式
> 出力先: docs/api/以下（ファイル名はルートファイルと対応させる）
>
> 各ファイルは独立して処理できるため、
> サブエージェントを使って並列で生成してください。
```

## 並列処理のベストプラクティス

### 独立性の確認

並列処理に適したタスクは、互いに依存関係がないものです。

```bash
# 並列処理に適している（独立したファイル）
タスクA: src/components/Header.tsx の修正
タスクB: src/components/Footer.tsx の修正
タスクC: src/components/Sidebar.tsx の修正

# 並列処理に不向き（依存関係あり）
タスクA: 型定義ファイルの作成（タスクBが依存）
タスクB: APIクライアントの実装（タスクAの型を使用）
タスクC: コンポーネントの実装（タスクBのAPIクライアントを使用）
```

### 適切なタスクサイズ

サブエージェントに与えるタスクは、適切なサイズに分割します。

```bash
# 大きすぎるタスク（一つのサブエージェントで処理しにくい）
> 全てのReactコンポーネントをリファクタリングしてください

# 適切なサイズのタスク
> src/components/forms/以下の10ファイルを3つのグループに分け、
> 各グループを並列でリファクタリングしてください
```

### 結果の統合を考慮した設計

並列処理後に親エージェントが結果を統合しやすいよう、出力形式を揃えます。

```bash
> 以下の3つのAPIエンドポイントを並列で分析してください。
> 各エンドポイントについて、以下の形式でレポートしてください:
>
> ## [エンドポイント名]
> - パフォーマンス: [良い/普通/悪い] - [理由]
> - セキュリティ: [良い/普通/悪い] - [理由]
> - 改善提案: [具体的な提案]
```

## まとめ

並列処理はサブエージェントの最も強力な活用方法の一つです。

- 独立したタスクを並列に実行することで処理時間を大幅に短縮できる
- 「並列」「同時」「並行」などのキーワードで明示的に指示する
- タスクの独立性を確認してから並列化を検討する
- 出力形式を統一することで結果の統合を容易にする
- エージェントタイプの特性を理解して適切に使い分ける
