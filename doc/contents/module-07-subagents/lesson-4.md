# 実践：大規模タスクの分解

## はじめに

これまでのレッスンでサブエージェントの基本、並列処理のパターン、worktreeとの連携を学びました。このレッスンでは、実際の開発現場で直面するような大規模タスクを、サブエージェントを使って効率的に分解・実行する実践的な手法を学びます。

## タスク分解の原則

大規模なタスクをサブエージェントで処理するには、適切な分解が鍵となります。

### 分解の4つの基準

**1. 独立性（Independence）**
各サブタスクが他のサブタスクの完了を待たずに実行できるか。

**2. 完結性（Completeness）**
各サブタスクが明確な完了条件を持っているか。

**3. 均一性（Uniformity）**
サブタスク間の作業量がほぼ均等か（一部が過度に重くならないか）。

**4. 検証可能性（Verifiability）**
各サブタスクの結果が独立して検証できるか。

```
良い分解例:
メインタスク: 10のAPIエンドポイントに対してテストを作成する
  └ サブタスク1: /api/users のテスト（独立・完結・検証可能）
  └ サブタスク2: /api/products のテスト（独立・完結・検証可能）
  └ サブタスク3: /api/orders のテスト（独立・完結・検証可能）
  ...

悪い分解例:
メインタスク: 認証システムを実装する
  └ サブタスク1: ユーザーモデルを作成（サブタスク2が依存）
  └ サブタスク2: JWTミドルウェアを実装（サブタスク1に依存）
  ※ これは並列化できないため直列で行うべき
```

## 実践例1: 複数ファイルにまたがるリファクタリング

### シナリオ

既存のコードベースで、コールバックパターンで書かれた非同期処理を async/await に移行する作業を行います。

### ステップ1: 調査フェーズ（並列）

```bash
> 以下を並列で調査してください:
>
> 調査1: src/services/以下でコールバックパターンが使われているファイルをすべてリストアップ
> 調査2: src/controllers/以下で同様にコールバックパターンを使っているファイルをリストアップ
> 調査3: src/middleware/以下で同様に調査
> 調査4: src/utils/以下で同様に調査
>
> 各ファイルについて、コールバックを使っている関数名と行番号も記録してください。
> 調査結果は統合してまとめてください。
```

### ステップ2: 移行計画の策定

```bash
> 調査結果を基に移行計画を作成してください:
>
> 1. ファイルを依存関係に基づいてグループ分け
>    （他のファイルから呼び出されるutilsは最初に移行する等）
> 2. 各グループを並列処理できる単位に分割
> 3. 移行順序と並列化の戦略を明確にする
> 4. リスクの高い箇所を特定（テストがない、複雑な処理等）
```

### ステップ3: worktreeでの並列移行

```bash
> "async-migration"worktreeを作成して、以下のグループを並列で移行してください:
>
> グループ1（ユーティリティ、最初に移行）:
> - src/utils/fileHelper.js: readFile, writeFile関数
> - src/utils/dbHelper.js: query, transaction関数
>
> グループ2（サービス層、グループ1完了後に並列実行）:
> - src/services/userService.js: getUser, createUser, updateUser
> - src/services/productService.js: getProducts, updateInventory
> - src/services/orderService.js: createOrder, processOrder
>
> 各ファイルの移行後、既存のテストが通ることを確認してください。
```

### ステップ4: テストと検証

```bash
> 移行完了後、以下を実行してください:
>
> 1. npm test を実行して全テストが通ることを確認
> 2. 移行されたコードに対してコードレビューを実施
>    （async/awaitの正しい使い方、エラーハンドリングの適切さ）
> 3. パフォーマンステスト（移行前後で応答時間を比較）
> 4. 移行レポートを作成（変更ファイル数、関数数、テスト結果）
```

## 実践例2: コードベースマイグレーション

### シナリオ

モノリシックなExpressアプリケーションをマイクロサービスアーキテクチャに分割します。

### フェーズ1: 現状分析（並列）

```bash
> 現在のモノリス構造を並列で分析してください:
>
> 分析1: src/routes/以下の全エンドポイントをリストアップし、
>         ドメイン（ユーザー管理、商品管理、注文管理等）で分類する
>
> 分析2: src/models/以下の全モデルをリストアップし、
>         各ドメインとの関連性をマッピングする
>
> 分析3: src/services/以下の全サービスをリストアップし、
>         ドメイン横断的な依存関係を特定する
>
> 分析4: データベーススキーマを分析し、
>         マイクロサービス化した場合の分割案を作成する
>
> 全分析完了後、マイクロサービス分割の提案をまとめてください。
```

### フェーズ2: 各マイクロサービスの並列構築

```bash
> 以下のマイクロサービスを並列で構築してください:
>
> サービス1: user-service（ポート3001）
> - ディレクトリ: services/user-service/
> - 担当API: /users, /auth
> - 担当モデル: User, Session
> - 独自のデータベース接続設定
>
> サービス2: product-service（ポート3002）
> - ディレクトリ: services/product-service/
> - 担当API: /products, /categories, /inventory
> - 担当モデル: Product, Category
> - 独自のデータベース接続設定
>
> サービス3: order-service（ポート3003）
> - ディレクトリ: services/order-service/
> - 担当API: /orders, /cart
> - 担当モデル: Order, Cart
> - user-serviceとproduct-serviceへのHTTPクライアント
>
> 各サービスは独立してテスト可能な状態で作成してください。
```

## 実践例3: テスト生成の大規模展開

### シナリオ

テストカバレッジが低いプロジェクト全体にテストを追加します。

### 現状分析

```bash
> テストカバレッジの現状を分析してください:
>
> 1. npm run test:coverage を実行してカバレッジレポートを生成
> 2. カバレッジが50%未満のファイルをリストアップ
> 3. ファイルをテストの複雑さでランク付け:
>    - 高: ビジネスロジックが多い（優先度高）
>    - 中: ユーティリティ関数（優先度中）
>    - 低: 設定ファイルや定数（優先度低）
> 4. 優先度の高いものから順に、テスト追加の計画を立てる
```

### 並列テスト生成

```bash
> 以下のファイルグループに対してテストを並列で生成してください:
>
> グループA（コアビジネスロジック - 最優先）:
> - src/services/paymentService.ts
>   テスト要件: 決済フロー全体、エラーケース（カード拒否、残高不足）、
>   リトライロジック
> - src/services/inventoryService.ts
>   テスト要件: 在庫チェック、在庫更新、同時アクセス時の整合性
>
> グループB（APIレイヤー）:
> - src/controllers/userController.ts
>   テスト要件: 全エンドポイントの正常系・異常系、バリデーション
> - src/controllers/orderController.ts
>   テスト要件: 注文作成フロー、状態遷移、権限チェック
>
> グループC（ユーティリティ）:
> - src/utils/dateFormatter.ts
>   テスト要件: 各フォーマット関数、エッジケース（うるう年、タイムゾーン）
> - src/utils/validator.ts
>   テスト要件: 全バリデーション関数、境界値テスト
>
> 各グループを並列で処理し、テスト作成後にnpm testで動作確認してください。
```

## タスク分解のテンプレート

実際の開発で使えるタスク分解テンプレートです。

### 分解テンプレート1: ファイル単位の並列処理

```bash
> 以下の[タスク名]を並列で実行してください:
>
> 対象ファイル群:
> - [ファイルパス1]: [具体的な作業内容]
> - [ファイルパス2]: [具体的な作業内容]
> - [ファイルパス3]: [具体的な作業内容]
>
> 各ファイルは独立して処理できます。
> 完了条件: [何をもって完了とするか]
> 報告形式: [結果をどのような形式で報告するか]
```

### 分解テンプレート2: フェーズ別処理

```bash
> [大規模タスク]を以下のフェーズで実行してください:
>
> フェーズ1（並列実行可能）:
> - サブタスク1a: [内容]
> - サブタスク1b: [内容]
> - サブタスク1c: [内容]
>
> フェーズ2（フェーズ1完了後、並列実行可能）:
> - サブタスク2a: [内容]（サブタスク1aの結果を使用）
> - サブタスク2b: [内容]（サブタスク1bの結果を使用）
>
> フェーズ3（全フェーズ完了後）:
> - 統合テスト実行
> - レポート生成
```

### 分解テンプレート3: ドメイン別並列処理

```bash
> [プロジェクト全体のタスク]を、以下のドメインに分けて並列処理してください:
>
> ドメイン1（[担当領域]）:
> 担当ファイル: [パターン、例: src/user/**]
> 作業内容: [具体的な作業]
> 完了条件: [条件]
>
> ドメイン2（[担当領域]）:
> 担当ファイル: [パターン]
> 作業内容: [具体的な作業]
> 完了条件: [条件]
>
> 各ドメインを独立したサブエージェントで処理してください。
```

## サブエージェント活用のベストプラクティス

### ベストプラクティス1: 明確な完了条件を定義する

```bash
# 曖昧な指示（避けるべき）
> コードをリファクタリングしてください

# 明確な完了条件付きの指示（推奨）
> 以下の条件が全て満たされたらリファクタリング完了とします:
> 1. 全ての関数の行数が30行以内
> 2. 重複するコードが排除されている
> 3. 既存のテストが全てパス
> 4. TypeScriptの型エラーがゼロ
> この条件を満たすようにリファクタリングしてください。
```

### ベストプラクティス2: 進捗報告のフォーマットを統一する

```bash
> 各サブタスクが完了したら、以下のフォーマットで報告してください:
>
> ## [サブタスク名] 完了
> - 処理ファイル数: X件
> - 変更行数: 追加X行 / 削除X行
> - テスト結果: X/X通過
> - 問題点（あれば）: [内容]
> - 次のアクション（あれば）: [内容]
```

### ベストプラクティス3: エラーハンドリングを事前に定義する

```bash
> 作業中に以下の状況が発生した場合の対応を事前に定義します:
>
> - 構文エラーが発生した場合: エラー内容を記録して次のファイルに進む
> - テストが失敗した場合: 変更を保留してメインエージェントに報告
> - 依存関係の問題が発生した場合: 作業を中断してメインエージェントに確認
>
> この対応方針に従って、並列処理を進めてください。
```

### ベストプラクティス4: 適切な粒度を見極める

```bash
# 粒度が細かすぎる（オーバーヘッドが大きい）
> 各クラスのメソッド1つずつを別々のサブエージェントで修正してください
# → 1000個のサブエージェントが起動する可能性

# 適切な粒度
> src/以下のファイルを10個ずつのグループに分けて、
> 各グループを並列処理してください
# → 数個〜数十個のサブエージェントが適切に機能する
```

### ベストプラクティス5: 結果の統合方法を事前に決める

```bash
> 並列処理完了後の統合フェーズ:
>
> 1. 各サブエージェントの結果をcollected-results.mdに集約
> 2. 矛盾や競合がないか確認
> 3. 全体のテストスイートを実行
> 4. 最終レポートを生成
>
> この統合プロセスを計画に含めてください。
```

## よくある失敗パターンと対策

### 失敗1: 依存関係を無視した並列化

```bash
# 問題のある指示
> 以下を並列で実装してください:
> - TypeScriptの型定義（types/api.ts）
> - APIクライアント（api/client.ts）←型定義に依存
> - コンポーネント（components/UserList.tsx）←APIクライアントに依存

# 修正後（直列化）
> 以下を順番に実装してください:
> 1. まず types/api.ts の型定義を完成させる
> 2. 完了後、api/client.ts を実装
> 3. 完了後、components/UserList.tsx を実装
```

### 失敗2: コンテキスト不足のサブエージェント指示

```bash
# コンテキスト不足（問題あり）
> エラーを修正してください
# → サブエージェントはどんなエラーか不明

# コンテキスト付き（正しい）
> src/api/orders.ts の createOrder 関数で発生する以下のエラーを修正してください:
> TypeError: Cannot read properties of undefined (reading 'userId')
> エラーは注文作成時にユーザーオブジェクトがnullの場合に発生します。
> 該当ファイルを読み込んで、nullチェックを追加してください。
```

### 失敗3: 結果の統合を忘れる

```bash
# 統合フェーズを含めた完全な指示
> 以下を並列で実行してください。
> 全て完了後、必ず統合フェーズを実行してください:
>
> [並列タスク...]
>
> 統合フェーズ:
> - 全ての変更ファイルをgit statusで確認
> - npm test で全テストを実行
> - 変更点の一覧をCHANGELOG形式でまとめる
```

## まとめ

大規模タスクをサブエージェントで効率的に分解・実行するためのポイントを整理します。

- タスク分解は独立性・完結性・均一性・検証可能性の4基準で判断する
- 依存関係のあるタスクは直列、独立したタスクは並列に分解する
- 明確な完了条件と報告フォーマットを事前に定義する
- 適切な粒度（ファイル単位〜ディレクトリ単位）でサブエージェントを配置する
- エラー発生時の対応方針を事前に定義しておく
- 並列処理後の統合フェーズを忘れずに計画に含める

サブエージェントを使いこなすことで、人間では何時間もかかる大規模作業を、Claude Codeが効率的かつ安全に自動化できます。これらの技術はClaude Codeを真のパワーツールとして活用するための核心スキルです。
