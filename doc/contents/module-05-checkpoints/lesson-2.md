# リワインド操作

## はじめに

前のレッスンでチェックポイントの基本を学びました。このレッスンでは、チェックポイントを使って過去の状態に戻す**リワインド操作**を詳しく学びます。全ての変更を元に戻す完全リワインドから、特定のファイルだけを戻す部分リワインドまで、様々なシナリオに対応できるようになりましょう。

## リワインドの基本操作

リワインドとは、チェックポイントが記録した時点の状態にファイルを復元する操作です。

```bash
# Claude Codeでリワインドを依頼する基本的な方法
> 最後のチェックポイントにリワインドして

# または特定のチェックポイントIDを指定
> cp_002にリワインドして

# または日本語で状況を説明
> さっきHeader.tsxを編集する前の状態に戻して
```

### リワインドの確認プロセス

リワインドを実行すると、Claude Codeは変更内容を表示して確認を求めます:

```
リワインド確認:
──────────────────────────────────────────────
チェックポイント: cp_002 (14:45:10)
説明: APIクライアントのリファクタリング前

以下のファイルが変更前の状態に戻ります:
  • src/api/client.ts
  • src/api/types.ts

変更の差分 (src/api/client.ts):
- const apiClient = axios.create({
-   baseURL: import.meta.env.VITE_API_URL,
-   timeout: 5000,
- });
+ const apiClient = axios.create({
+   baseURL: process.env.REACT_APP_API_URL,
+   timeout: 3000,
+ });

リワインドを実行しますか？ [y/N]:
```

確認後にリワインドが実行されます:

```
リワインド完了: cp_002 の状態に復元しました
復元されたファイル:
  ✓ src/api/client.ts
  ✓ src/api/types.ts
```

## 完全リワインドと部分リワインド

### 完全リワインド

チェックポイント時点で変更された**全てのファイル**を元の状態に戻します。

```bash
# 最新のチェックポイントに完全リワインド
> 最後のチェックポイントに戻して

# 特定のチェックポイントに完全リワインド
> cp_001の状態に全部戻して
```

完全リワインドが適しているケース:
- 変更全体が期待と異なった場合
- 複数のファイルが連動して誤った変更を受けた場合
- 最初からやり直した方が効率的な場合

### 部分リワインド

チェックポイント時点の変更のうち、**特定のファイルだけ**を元の状態に戻します。

```bash
# 特定のファイルだけリワインド
> src/api/client.tsだけ cp_002 の状態に戻して

# 複数の特定ファイルをリワインド
> src/api/client.ts と src/api/types.ts を
> cp_002の状態に戻して。他のファイルはそのままで。
```

部分リワインドが適しているケース:
- 一部の変更は良かったが、特定のファイルだけ問題があった場合
- 複数ファイルの変更のうち、一部だけ採用したい場合

```bash
# 実際の使用例
> 今回の変更でcomponents/Button.tsxの変更は良かったけど、
> components/Header.tsxの変更だけ元に戻して

# Claude Codeの応答例
# 確認: Header.tsx のみを cp_003 の状態に戻します
# Button.tsx の変更は維持されます
# よろしいですか？
```

## リワインド前の変更確認

リワインドを実行する前に、変更内容を詳しく確認することが重要です。

### diff 形式での確認

```bash
# リワインド前に差分を確認
> cp_001からの変更内容を差分で見せて

# Claude Codeが表示する差分の例
```

```diff
--- src/utils/formatDate.ts (cp_001 時点)
+++ src/utils/formatDate.ts (現在)
@@ -1,12 +1,18 @@
 import { format } from 'date-fns';
+import { utcToZonedTime } from 'date-fns-tz';

 export function formatDate(date: Date): string {
-  return format(date, 'yyyy/MM/dd');
+  const timeZone = 'Asia/Tokyo';
+  const zonedDate = utcToZonedTime(date, timeZone);
+  return format(zonedDate, 'yyyy/MM/dd HH:mm');
 }
```

### ファイルごとの詳細確認

```bash
# 特定のファイルの変更内容を確認
> 今回変更されたsrc/api/client.tsの内容を見せて

# 現在のコードと変更前のコードを並べて確認
> cp_002時点のsrc/api/client.tsと現在の内容を比較して
```

### 変更内容の評価基準

リワインドするかどうかを判断する際のチェックポイント:

```
変更を維持する場合:
  ✓ テストが全て通っている
  ✓ コードが意図した動作をしている
  ✓ 可読性が向上している
  ✓ パフォーマンスが改善されている

リワインドを検討する場合:
  ✗ テストが失敗している
  ✗ 予期しないバグが発生した
  ✗ コードが複雑になりすぎた
  ✗ 変更の目的とは異なる部分まで修正されている
```

## リワインド vs git reset

リワインドと`git reset`は似た目的を持ちますが、使い分けが重要です。

### リワインドを使うべき場面

```bash
# Claude Codeのセッション内での試行錯誤
> このアルゴリズムを最適化してみて

# 最適化の結果が悪かった場合
> さっきの変更前に戻して
```

リワインドが適している理由:
- **gitの履歴を汚さない**（未コミットの変更への対応）
- **高速**（シンプルなファイル復元）
- **自然言語で指定できる**（「さっきの状態に戻して」）
- **gitを意識せずに安全に試せる**

### git reset を使うべき場面

```bash
# コミット済みの変更を取り消す場合
git reset --soft HEAD~1   # 直前のコミットを取り消し（変更は保持）
git reset --hard HEAD~1   # 直前のコミットを完全に取り消し

# 特定のコミットまで戻る
git reset --hard abc1234
```

`git reset`が適している理由:
- **コミット履歴の操作**が必要な場合
- **チームと共有した後**の変更を戻す場合
- **ブランチの操作**が必要な場合

### 使い分けの判断フロー

```
変更はgitにコミット済み？
  ├── Yes → git reset を検討
  │          ├── リモートにプッシュ済み？
  │          │     ├── Yes → git revert を使う（履歴を保持）
  │          │     └── No  → git reset --soft/hard を使う
  │
  └── No → チェックポイントのリワインドを使う
           ├── Claude Codeのセッション内？ → リワインド
           └── セッション外？ → git checkout でファイル復元
```

## 実践的なリワインドシナリオ

### シナリオ1: リファクタリングが失敗した場合

```bash
# リファクタリングを依頼
> src/services/authService.tsをクラスベースから
> 関数ベースにリファクタリングして

# リファクタリング後、テストを実行
> テストを実行して

# テストが失敗した場合
# テスト結果:
# FAIL src/services/authService.test.ts
#   ✗ ログイン処理が正常に動作しない
#   ✗ トークンの更新が機能しない

# リワインドを実行
> テストが失敗したので、authService.tsのリファクタリング前に戻して

# Claude Codeの応答
# cp_003（authService.tsの変更前）にリワインドします
# ✓ src/services/authService.ts を復元しました
```

### シナリオ2: 複数ファイルの変更で一部だけ問題がある場合

```bash
# 複数コンポーネントの更新を依頼
> ユーザーインターフェースを刷新して。
> Header、Sidebar、Footerを新しいデザインシステムに合わせて更新して

# 変更後の確認
> 変更されたファイルの一覧を見せて

# 一部に問題があることを発見
# HeaderとSidebarは良いが、Footerに問題がある
> Footerだけ変更前の状態に戻して。
> HeaderとSidebarの変更は維持して。

# 部分リワインドが実行される
# ✓ src/components/Footer.tsx を cp_005 の状態に復元しました
# src/components/Header.tsx は変更を維持します
# src/components/Sidebar.tsx は変更を維持します
```

### シナリオ3: 段階的なリワインドによる問題の特定

```bash
# バグの原因を特定するためのリワインド
> ビルドエラーが発生しています。原因を特定するため、
> 一つ前のチェックポイントに戻してビルドが通るか確認して

# チェックポイントごとに確認しながら絞り込む
> さらに一つ前のcp_002に戻してビルドが通るか確認して

# バグが発生したチェックポイントを特定したら
> cp_003の変更内容を詳しく分析して、問題のある箇所を教えて
```

## まとめ

リワインド操作は、Claude Codeを使った開発の安全網として機能します。完全リワインドで全ての変更を取り消したり、部分リワインドで特定のファイルだけを元に戻したりと、柔軟な復元が可能です。リワインドは未コミットの変更への対応に特化しており、gitの履歴を汚さずに試行錯誤できるのが大きなメリットです。次のレッスンでは、チェックポイントとリワインドを活用した実験的な開発手法について学びます。
