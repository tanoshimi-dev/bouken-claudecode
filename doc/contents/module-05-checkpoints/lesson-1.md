# チェックポイントの基本

## はじめに

Claude Codeを使って開発を進める際、AIが行う変更が予期しない結果をもたらすことがあります。そのような場面で安心して作業を続けられるのが**チェックポイント機能**です。このレッスンでは、チェックポイントが何であるか、どのように機能するか、そしてgitコミットとの違いについて詳しく学びます。

## チェックポイントとは

チェックポイントは、Claude Codeがファイルを変更する**前**に自動的に保存されるスナップショットです。作業状態の「セーブポイント」と考えると分かりやすいでしょう。

```
変更前の状態 ──[チェックポイント保存]──> Claude Codeがファイルを編集 ──> 変更後の状態
                        ↑
                   自動で記録される
```

チェックポイントが保存される主なタイミング:

- Claude Codeがファイルを**新規作成**する前
- Claude Codeが既存ファイルを**編集**する前
- Claude Codeが複数のファイルを**まとめて変更**する前
- 長い会話の中で**定期的**に

## チェックポイントの自動作成

Claude Codeはファイルへの書き込み操作を行う前に、自動的にチェックポイントを作成します。これはユーザーが何もしなくても、バックグラウンドで行われます。

```bash
# Claude Codeを起動して作業を開始
claude

# 以下のような指示を出すと、実行前にチェックポイントが作成される
> src/components/Header.tsxをリファクタリングして
```

Claude Codeの実行ログにはチェックポイントの作成が表示されます:

```
Checkpoint created: cp_20240315_143022
Editing src/components/Header.tsx...
```

### 自動作成されるタイミングの詳細

| 操作 | チェックポイント作成 |
|------|---------------------|
| ファイルの編集 | 編集前に自動作成 |
| ファイルの新規作成 | 作成前に自動作成 |
| ファイルの削除 | 削除前に自動作成 |
| 複数ファイルの一括変更 | 変更前に自動作成 |
| コマンド実行のみ | 作成されない |

## チェックポイント履歴の確認

現在のセッションで作成されたチェックポイントの一覧を確認できます。

```bash
# チェックポイント一覧を表示するよう依頼
> チェックポイントの履歴を見せて
```

または、Claude Code内で専用のスラッシュコマンドを使用します:

```
/checkpoints
```

表示される情報の例:

```
チェックポイント履歴:
──────────────────────────────────────────────
cp_001  14:30:22  Header.tsxの編集前
        変更ファイル: src/components/Header.tsx

cp_002  14:45:10  APIクライアントのリファクタリング前
        変更ファイル: src/api/client.ts
                     src/api/types.ts

cp_003  15:02:33  テストファイルの追加前
        変更ファイル: src/components/Header.test.tsx
──────────────────────────────────────────────
最新のチェックポイント: cp_003
```

各チェックポイントには以下の情報が含まれます:

- **ID**: 一意の識別子（例: `cp_001`）
- **タイムスタンプ**: 作成された日時
- **説明**: どの操作の前に作成されたか
- **変更ファイル**: 影響を受けたファイルの一覧

## チェックポイントとgitコミットの違い

チェックポイントとgitコミットは似ているようで、全く異なる目的を持つ仕組みです。

### gitコミット

```bash
# gitコミットは開発者が明示的に作成する
git add src/components/Header.tsx
git commit -m "feat: Headerコンポーネントをリファクタリング"
```

gitコミットの特徴:
- **手動**で作成する
- リポジトリの**永続的な履歴**として残る
- チームメンバーと**共有**できる
- `git log`で確認可能
- プッシュして**リモートに反映**できる

### チェックポイント

```
# チェックポイントはClaude Codeが自動で作成する
# ユーザーは何もする必要がない
```

チェックポイントの特徴:
- **自動**で作成される
- **セッション内**で有効（通常はセッション終了後に破棄）
- gitの作業ツリーに**影響しない**
- チームとは**共有されない**
- 細かい粒度で**安全網**を提供する

### 具体的な違いの比較

| 比較項目 | チェックポイント | gitコミット |
|---------|----------------|-------------|
| 作成タイミング | 自動（変更前） | 手動 |
| 保存期間 | セッション内 | 永続 |
| 共有 | 不可 | 可能 |
| 粒度 | 細かい（操作ごと） | 粗い（意味のある単位） |
| 目的 | 実験・安全網 | 変更履歴の記録 |
| ブランチ | なし | あり |

## 実践例：チェックポイントを意識した作業フロー

以下は実際の開発でチェックポイントがどう機能するかの具体例です。

```bash
# 1. Claude Codeを起動
claude

# 2. 既存コードの変更を依頼（この時点でチェックポイントが自動作成される）
> src/utils/formatDate.tsのformatDate関数を
> タイムゾーン対応に改修してください

# Claude Codeの実行ログ（内部動作）:
# Checkpoint cp_001 created
# Reading src/utils/formatDate.ts...
# Modifying src/utils/formatDate.ts...
# Done.

# 3. 結果を確認
> 変更内容を確認させて

# 4. 問題があればリワインド（次のレッスンで詳しく学びます）
> cp_001にリワインドして
```

## チェックポイントが保護してくれる場面

チェックポイントは特に以下のような場面で役立ちます:

**意図しない変更への対応**

```bash
> このコンポーネント全体をTypeScriptに書き換えて
# → 変換後のコードが期待と違った場合でも、チェックポイントから戻れる
```

**実験的な変更のテスト**

```bash
> パフォーマンス改善のため、このロジックをメモ化してみて
# → パフォーマンスが改善しなかった場合、元の状態に戻せる
```

**複数ファイルにまたがる変更**

```bash
> APIのレスポンス形式を変更して、それに合わせて全コンポーネントを更新して
# → 更新後に問題が発覚した場合、一括で元に戻せる
```

## まとめ

チェックポイントは、Claude Codeが変更を加える前に自動的に作成されるセーフティネットです。開発者が意識しなくても常に保護されているため、安心してAIによる変更を試みることができます。gitコミットとは異なり、セッション内の細かい操作単位で保存されるのが特徴です。次のレッスンでは、チェックポイントを使って過去の状態に戻す「リワインド操作」を学びます。
