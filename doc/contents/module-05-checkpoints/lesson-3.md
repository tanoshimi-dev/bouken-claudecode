# 安全な実験的開発

## はじめに

チェックポイントとリワインドの仕組みを理解した今、これらを最大限に活用した**実験的開発**のアプローチを学びましょう。「失敗しても戻せる」という安心感があることで、大胆に新しいアプローチを試せるようになります。このレッスンでは、チェックポイントを活用した開発手法と、gitブランチ戦略との組み合わせを実践的に学びます。

## 実験的開発とは

実験的開発とは、**複数のアプローチを安全に試しながら最良の解決策を探す**開発スタイルです。

従来の開発アプローチ（チェックポイントなし）:

```
慎重に計画 → 実装 → 問題発覚 → 手動で元に戻す → 別のアプローチ
                                    ↑
                              時間と労力が必要
```

チェックポイントを活用した実験的開発:

```
アイデアA を試す → うまくいかない → リワインド → アイデアB を試す
                       ↑                ↑
                   自動チェックポイント  一瞬で戻れる
```

## 複数アプローチの安全な試行

### アプローチ比較の実践例

```bash
# Claude Codeを起動
claude

# アプローチAを試す
> パフォーマンス改善のため、ユーザーリストのフィルタリングロジックを
> useMemoを使ってメモ化してみてください

# Claude Codeが自動でチェックポイントを作成してから変更を適用
# チェックポイント: cp_001（メモ化前の状態）

# パフォーマンスを測定
> React DevToolsで計測した結果、レンダリング時間が 120ms → 95ms になりました。
> さらに改善できますか？

# アプローチBを試す（チェックポイントcp_001を起点にできる）
> 現在の実装に加えて、仮想スクロール（react-window）も試してみてください

# うまくいかなかった場合はリワインド
> react-windowの導入が複雑すぎました。cp_002の状態（useMemoのみ）に戻してください
```

### 並行したアプローチの比較

```bash
# アプローチ1: REST API
> ユーザーデータの取得をREST APIで実装してください
# チェックポイント: cp_rest_before

# テストと評価
> 実装を確認しました。次にGraphQLで同じ機能を実装したらどうなるか比較したいです

# cp_rest_beforeにリワインドしてアプローチ2を試す
> cp_rest_beforeに戻してから、GraphQL版を実装してみてください
# チェックポイント: cp_graphql_before（REST版に戻した後の状態）

# 比較結果を評価してどちらを採用するか決定
> 両方の実装を比較して、このプロジェクトに適しているのはどちらか教えてください
```

## チェックポイントのベストプラクティス

### 1. 意味のある単位でチェックポイントを意識する

Claude Codeは自動でチェックポイントを作成しますが、**重要な節目では意識的に確認**しておくことが重要です。

```bash
# 重要な変更の前に現在の状態を確認
> 現在のチェックポイントIDを教えてください

# 出力例
# 現在の最新チェックポイント: cp_007 (15:23:41)
# 変更ファイル数: 3

# この情報をメモしておくことで、後から確実に戻れる
```

### 2. 段階的な変更で細かいチェックポイントを活用する

大きな変更は分割して段階的に行うことで、問題が発生したときに最小限のロールバックで済みます。

```bash
# 悪い例: 全てを一度に変更
> 認証システム全体をJWTからOAuthに移行してください

# 良い例: 段階的に変更
> まず認証の型定義だけ更新してください
# → チェックポイントcp_010

> 次にAPIクライアントの認証ヘッダー処理を更新してください
# → チェックポイントcp_011

> 最後にログインフォームの送信処理を更新してください
# → チェックポイントcp_012
```

### 3. リワインドのコストを意識した作業順序

```bash
# 影響範囲の小さい変更から始める
# 1. 型定義（他のファイルへの影響が最も小さい）
> src/types/user.tsのUserProfile型を更新してください

# 2. ユーティリティ関数（独立性が高い）
> src/utils/userHelpers.tsを更新してください

# 3. サービス層（コンポーネントへの影響が少ない）
> src/services/userService.tsを更新してください

# 4. コンポーネント（最後に変更）
> src/components/UserProfile.tsxを更新してください
```

### 4. チェックポイントをコミュニケーションの道具にする

```bash
# チェックポイントIDをメモして、作業の文脈を保持する
> 現在のチェックポイントIDを記録しておいて。
> これを「認証機能着手前」として参照できるようにして

# 後から参照
> 「認証機能着手前」のチェックポイントに戻して
```

## チェックポイントとgitブランチ戦略の組み合わせ

チェックポイントとgitブランチを組み合わせることで、より強力な開発ワークフローを構築できます。

### 推奨ワークフロー

```
main ──────●──────────────────────────●────────▶
            └── feature/new-auth ──────┘
                     │
                     │ (Claude Codeの作業範囲)
                     │
                  cp_001 → cp_002 → cp_003
                     │         │
                     │    問題発覚・リワインド
                     │
                  cp_004 → コミット → PRマージ
```

### 実際のワークフロー

```bash
# 1. gitブランチを作成（大きな変更の境界）
git checkout -b feature/user-profile-redesign

# 2. Claude Codeで実験的な開発を開始
claude

# 3. チェックポイントを活用して試行錯誤
> ユーザープロフィールページのレイアウトを新しいデザインに変更して
# → チェックポイント cp_001 自動作成

> レイアウトはいい感じです。次にアバター画像のアップロード機能を追加して
# → チェックポイント cp_002 自動作成

> アップロード機能に問題があります。cp_002に戻してやり直しましょう
# → cp_002 にリワインド

> アバター画像のアップロードをより単純な方法で実装して
# → チェックポイント cp_003 自動作成

# 4. 満足のいく状態になったらgitコミット
git add src/components/UserProfile.tsx src/services/uploadService.ts
git commit -m "feat: ユーザープロフィールのリデザインとアバター機能"

# 5. PRを作成してレビュー依頼
git push origin feature/user-profile-redesign
```

### チェックポイントとブランチの役割分担

| 役割 | チェックポイント | gitブランチ |
|------|----------------|-------------|
| 試行錯誤の安全網 | 担当 | 担当しない |
| 機能単位の分離 | 担当しない | 担当 |
| チームとの共有 | 担当しない | 担当 |
| レビュープロセス | 担当しない | 担当 |
| 長期的な履歴管理 | 担当しない | 担当 |

## 実世界のワークフロー例

### 例1: 新機能の実装と最適化

```bash
# フィーチャーブランチを作成
git checkout -b feature/search-functionality

# Claude Codeで実装開始
claude

# フェーズ1: 基本実装
> 商品の全文検索機能をElasticsearchを使って実装してください

# フェーズ2: パフォーマンス改善を試す
> 検索クエリのキャッシュを追加して、レスポンスタイムを改善してください
# → 改善効果が不十分だったらリワインド

# フェーズ3: 別のアプローチ
> キャッシュの代わりに、検索インデックスの最適化を試してください
# → より効果的だった場合は維持

# 完成したらコミット
git add .
git commit -m "feat: 商品全文検索機能の実装（インデックス最適化済み）"
```

### 例2: バグ修正の安全な検証

```bash
# バグ修正ブランチを作成
git checkout -b fix/login-timeout-issue

# 原因調査
> ログイン処理でタイムアウトが発生するバグがあります。
> src/services/authService.tsを調査して原因を教えてください

# 修正案1を試す
> セッションの有効期限管理に問題があるようです。
> トークンリフレッシュのロジックを修正してください
# → チェックポイント cp_001

# テストで検証
> この修正でバグが再現するかテストしてください

# 修正案2に切り替え（cp_001にリワインド後）
> 修正案1では解決しませんでした。cp_001に戻して、
> 今度はCookieの設定を見直すアプローチで試してください

# 解決したらコミット
git add src/services/authService.ts
git commit -m "fix: ログインタイムアウトの原因となるCookie設定を修正"
```

> ビルドエラーが3件あります。エラーメッセージを貼ります:

## まとめ

チェックポイントを活用した実験的開発は、「失敗のコスト」を大幅に下げることで、より大胆な改善と最適化を可能にします。重要なポイントをおさらいします:

- 大きな変更は**段階的**に行い、各ステップでチェックポイントを活用する
- **複数のアプローチ**を安全に比較できるのがチェックポイントの強み
- **gitブランチ**と組み合わせることで、実験と履歴管理を分離できる
- チェックポイントはセッション内の安全網、gitは長期的な履歴管理と役割を明確に分ける

チェックポイントとリワインドをマスターすることで、Claude Codeとの協働がより安全で効率的になります。恐れずに新しいアプローチを試し、問題があればすぐにリセットする。この開発スタイルがAI支援開発の真骨頂です。
